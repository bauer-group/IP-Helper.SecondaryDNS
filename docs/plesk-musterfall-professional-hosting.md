# Musterfall: Plesk-Primary `ns8.professional-hosting.com` an Secondary `ns1.professional-hosting.com` anbinden

Eigenstaendige Anleitung fuer die Aufnahme **eines** konkreten Plesk-
Primary in eine laufende Secondary-Infrastruktur. Die zugrundeliegende
NOTIFY/AXFR-Mechanik beschreibt [PRIMARY-SETUP.md](../PRIMARY-SETUP.md);
dieses Doc ist ohne Querverweise vollstaendig durchfuehrbar.

> **Geltungsbereich:** Operativer Standard-Vorgang "neuer Plesk-Primary
> kommt dazu". Pattern ist `n:1` — der Secondary `ns1` bleibt stabil,
> Primaries (`ns2`, `ns3`, ... `nsN`) wechseln/kommen/gehen. Dieses Doc
> zeigt den Vorgang am Beispiel `ns8`; fuer `ns9`, `ns10` etc. wird die
> Anleitung 1:1 mit getauschten Werten wiederholt.

## Architektur-Pattern

```text
                    .com Registry (Glue-Records)
                              |
              +---------------+----------------+
              |                                |
              v                                v
+-------------------------+      +-------------------------+
| Secondary (PowerDNS)    |      | Plesk-Primaries         |
|                         | <----+ ns2  (Plesk #2)         |
| ns1.professional-       |  N   + ns3  (Plesk #3)         |
| hosting.com             |  O   + ...                     |
|                         |  T   + ns8  (Plesk #8, neu)    |
| - SLAVE, stabil         |  I   + nsN  (Plesk #N)         |
| - aggregiert alle Zonen |  F   |                         |
| - read-only Mirror      |  Y   | - kommen + gehen        |
|                         | <----+ - jeder ist MASTER      |
| supermasters:           |      |   eigener Kundenzonen   |
|   (ns2-ip, ns2-host)    |      | - SOA-MNAME = eigener   |
|   (ns3-ip, ns3-host)    |      |   Hostname              |
|   (ns8-ip, ns8-host)    |      +-------------------------+
+-------------------------+
```

**Konsequenzen aus dem Pattern:**

- Der Secondary `ns1` ist **Slave** — pflegt keine Quelldaten, sondern
  zieht alle Zonen via AXFR aus den jeweiligen Plesks. Wird einmal mit
  `install.sh` installiert und nie mehr im Hostnamen / IP veraendert.
- Jeder Plesk ist **Master seiner Kundenzonen** und gibt sich auch im
  SOA-Record so aus: SOA-MNAME jeder Zone = Plesk-Hostname (z.B.
  `ns8.professional-hosting.com.` fuer Zonen auf Plesk #8).
- Die NS-Records **in jeder Zone** listen den jeweiligen Plesk-Primary
  **plus** `ns1` (zwei oeffentliche NS fuer Resilienz) — beides wird
  ueber das Plesk-DNS-Template ausgerollt.

## Szenario

| Rolle                              | Hostname                       | IPv4                | IPv6                   | Aenderbar?           |
|------------------------------------|--------------------------------|---------------------|------------------------|----------------------|
| **Secondary / Slave (existiert)**  | `ns1.professional-hosting.com` | _IPv4 ns1 - tragen_ | _IPv6 ns1 - tragen_    | nein, dauerhaft      |
| **Primary / Master (kommt jetzt)** | `ns8.professional-hosting.com` | `178.63.43.203`     | `2a01:4f8:120:538f::2` | ja (Plesk-Lifecycle) |

> Die IPs des Secondary `ns1` sind oben offen, weil sie im Auftrag nicht
> mitgeliefert wurden. Auf dem Secondary-Host ermitteln:
>
> ```bash
> ip -4 addr show scope global | awk '/inet /{print $2}'
> ip -6 addr show scope global | awk '/inet6 /{print $2}'
> ```

## Ist-Zustand (was Plesk aktuell ausliefert)

Plesk (`ns8`) generiert aktuell fuer Kundenzonen Zonefiles dieser Form
(Beispiel: `aschenbrenner-bau.de`):

```dns
; *** This file is automatically generated by Plesk ***
$TTL    86400

@   IN  SOA   ns1.professional-hosting.com. mb.meta-point.de. (
              2026052007 ; Serial
              10800      ; Refresh
              3600       ; Retry
              604800     ; Expire
              10800 )    ; Minimum

@   IN  NS    ns1.professional-hosting.com.
@   IN  NS    ns8.professional-hosting.com.
@   IN  A     178.63.43.203
@   IN  AAAA  2a01:4f8:120:538f::2
...
```

Drei Beobachtungen:

1. **SOA-MNAME zeigt auf `ns1`** — fachlich falsch. `ns1` ist der Slave
   (dieser PowerDNS), nicht der Master. Master ist Plesk selbst (`ns8`).
   Solange der MNAME auf `ns1` zeigt, hat die Welt einen Identity-
   Mismatch zwischen Datenursprung und SOA-Aussage.
2. **NS-Records `ns1` + `ns8`** — bereits korrekt. Beide Server werden
   als autoritativ angesagt.
3. **Hetzner-Default-NS im DNS-Zonen-Template** — noch nicht entfernt.
   Neu angelegte Zonen wuerden sonst weiterhin `ns1.first-ns.de.` etc.
   bekommen.

Diese Anleitung beseitigt (1) und (3). Punkt (2) ist bereits ok.

## Zielbild (was Plesk ausliefern soll)

Nach Abschluss aller Schritte gibt Plesk fuer Kundenzonen aus:

```dns
@   IN  SOA   ns8.professional-hosting.com. hostmaster.professional-hosting.com. (
              <serial> 10800 3600 604800 10800 )

@   IN  NS    ns8.professional-hosting.com.
@   IN  NS    ns1.professional-hosting.com.
```

Und auf Secondary `ns1` steht in der `supermasters`-Tabelle:

```text
(178.63.43.203,        ns8.professional-hosting.com)
(2a01:4f8:120:538f::2, ns8.professional-hosting.com)
```

PLUS die bereits existierenden Eintraege frueherer Plesks
(`ns2`..`ns7`), die jeweils mit ihrem eigenen Hostnamen als
Nameserver-Wert dort stehen.

## Schritt 1 - DNS-Zonen-Template auf Plesk (`ns8`) umbauen

`Tools & Einstellungen -> DNS-Zonen-Template`.

### 1a) Hetzner-Defaults entfernen

Drei NS-Eintraege markieren und `Entfernen`:

- [ ] `ns1.first-ns.de.`
- [ ] `robotns2.second-ns.de.`
- [ ] `robotns3.second-ns.com.`

### 1b) Eigene NS-Records hinzufuegen (Reihenfolge ist wichtig)

> **Plesk-Quirk:** Der SOA-MNAME ist **NICHT** im SOA-Template-UI
> einstellbar (Plesk zeigt dort nur TTL/Refresh/Retry/Expire/Minimum).
> Plesk leitet den MNAME stattdessen aus dem **zuerst angelegten NS-
> Record** ab (intern: NS-Record mit dem fruehesten `timestamp`, dann
> `id`). Konsequenz: **die Anlage-Reihenfolge der NS-Records
> entscheidet ueber den MNAME.**

Per `DNS-Eintrag hinzufuegen` **zwei** NS-Records anlegen — in dieser
Reihenfolge:

| # | Eintragstyp | Domain                      | Nameserver                      | Rolle              |
|---|-------------|-----------------------------|---------------------------------|--------------------|
| 1 | `NS`        | _leer (= `@`, Zone selbst)_ | `ns8.professional-hosting.com.` | **MNAME** (zuerst) |
| 2 | `NS`        | _leer (= `@`, Zone selbst)_ | `ns1.professional-hosting.com.` | weiterer NS        |

> Der trailing dot ist nicht kosmetisch — er macht den Namen absolut.
> Ohne ihn haengt Plesk bei der Zonenanlage die jeweilige Kundendomain
> dran (`ns8.professional-hosting.com.kunde-a.de.`) -> Bruch.
>
> Auf NS-Ebene sind beide Records gleichwertig — die Reihenfolge zaehlt
> NUR fuer die interne MNAME-Ableitung. DNS-Resolver behandeln NS als
> Set.

### 1c) SOA-Eintrags-Template setzen (nur Timer-Werte)

`SOA-Eintrags-Template` oeffnen. Wie im UI sichtbar gibt es hier nur
Timer/Format-Einstellungen, **keinen MNAME-Setter** — der MNAME stammt
aus Schritt 1b.

| Feld                | Wert                                |
|---------------------|-------------------------------------|
| TTL                 | `1 Tage`                            |
| Aktualisieren       | `3 Stunden`                         |
| Erneut versuchen    | `1 Stunden`                         |
| Ablauf              | `7 Tage`                            |
| Minimum-TTL         | `3 Stunden`                         |
| Seriennummernformat | `Von IETF und RIPE empfohlen` aktiv |

Plesk-Defaults sind sinnvoll und brauchen keine Anpassung. Wenn andere
NS-Service-Provider-Defaults gefordert sind (z.B. kuerzere Refresh-
Intervalle), hier setzen.

**RNAME (E-Mail)** wird in diesem UI ebenfalls nicht angeboten — Plesk
nimmt automatisch die Plesk-Server-Admin-E-Mail. Wer das pro Zone
aendern will: `Domains -> <domain> -> DNS-Einstellungen -> SOA-Record`.

### 1d) MNAME-Ableitung verifizieren

Nach Schritt 1b mit `dig` pruefen:

```bash
dig @178.63.43.203 aschenbrenner-bau.de SOA +short
# Erstes Token muss ns8.professional-hosting.com. sein (NICHT ns1.)
```

Wenn der MNAME trotzdem `ns1` ausspuckt, gilt einer der folgenden Faelle:

- Schritt 4 (Template auf bestehende Domains anwenden) noch nicht gelaufen
- NS-Records wurden in falscher Reihenfolge angelegt -> beide Records
  loeschen, `ns8` neu anlegen, danach `ns1` neu anlegen
- Pro-Zone-Override aktiv -> in der einzelnen Zone via
  `Domains -> <domain> -> DNS-Einstellungen -> Primary Name Server`
  manuell auf `ns8` umstellen

### 1e) Fallback: direkter DB-Eingriff (nicht supported)

Falls Reihenfolge-Trick aus 1b nicht greift (etwa weil Plesk die NS-
Records mit identischem Timestamp angelegt hat und nach `id` sortiert,
die je nach DB-State variieren kann), bleibt der DB-Weg:

```bash
# Auf dem Plesk-Server als root:
plesk db -e "SELECT id, host, displayHost, val, time_stamp \
             FROM dns_recs_t WHERE type='NS' ORDER BY time_stamp, id;"

# Wenn ns1 als erste Zeile erscheint, dessen time_stamp hochsetzen,
# damit ns8 davor rutscht (Werte vorher mit obigem SELECT verifizieren):
plesk db -e "UPDATE dns_recs_t \
             SET time_stamp = time_stamp + 1 \
             WHERE type='NS' AND val='ns1.professional-hosting.com.';"

# Cache invalidieren + bestehende Zonen neu synchronisieren:
plesk bin server_dns --update
```

> **Risiko:** `dns_recs_t` ist nicht Teil der Plesk-Public-API. Schema-
> Aenderungen in zukuenftigen Plesk-Releases (z.B. Spaltenumbenennungen,
> Migration auf neue Datenstruktur) koennen die Query brechen. Vor
> jedem Plesk-Major-Upgrade pruefen, ob der Workaround noch noetig ist
> — Plesk Ticket 126194 verfolgt einen offiziellen MNAME-Setter.

## Schritt 2 - Secondary als Slave bei Plesk eintragen

`Tools & Einstellungen -> Slave DNS Servers -> Hinzufuegen`.

| Feld         | Wert                       |
|--------------|----------------------------|
| IP-Adresse   | _IPv4 des Secondary `ns1`_ |
| IPv6-Adresse | _IPv6 des Secondary `ns1`_ |

Wirkung:

- Plesk traegt `ns1` in `allow-transfer` ein -> AXFR fuer `ns1` erlaubt
- Plesk sendet NOTIFY an diese IPs bei jeder Zonen-Aenderung auf `ns8`
- Plesk triggert NOTIFY initial fuer **alle bestehenden** Zonen auf
  `ns8`, sobald `ns1` neu eingetragen wird

> **Beide** IPs (v4 + v6) eintragen. PowerDNS prueft NOTIFY pro
> Source-IP einzeln gegen `supermasters`. IPv6-NOTIFY ohne IPv6-Eintrag
> in `supermasters` -> "no matching supermaster" -> Ignoriert.

## Schritt 3 - NS-Hostnamen ns1 und ns8 aufloesbar machen

Damit ein Resolver, der von einer Customer-Domain auf
`ns1.professional-hosting.com` bzw. `ns8.professional-hosting.com`
verwiesen wird, deren IPs ermitteln kann, muessen A/AAAA-Records fuer
diese Hostnamen existieren.

> **NICHT** in diesem Schritt: die NS-Delegation einer Domain
> umschalten. Das passiert getrennt in
> [Schritt 7](#schritt-7---ns-delegation-am-registrar-umstellen),
> erst nachdem `ns1` autoritativ antwortet
> ([Schritt 6](#schritt-6---funktion-verifizieren)).

### 3a) A/AAAA in der Zone `professional-hosting.com`

Wo immer `professional-hosting.com` selbst gehostet ist (Cloudflare,
Registrar-DNS, anderer Plesk, Anycast-Provider — **nicht zwingend
dieses Setup**), folgende Records anlegen oder pruefen:

```text
ns1   A     <IPv4 ns1>
ns1   AAAA  <IPv6 ns1>
ns8   A     178.63.43.203
ns8   AAAA  2a01:4f8:120:538f::2
```

Verifikation von aussen:

```bash
dig +short ns1.professional-hosting.com A AAAA
dig +short ns8.professional-hosting.com A AAAA
```

Wenn beide IPs zurueckkommen, ist Schritt 3 fertig.

### 3b) Hinweis: kein Glue noetig fuer Customer-Domains

Aus Sicht einer **Customer-Domain** (z.B. `aschenbrenner-bau.de`,
`example.com`, ...) sind `ns1.professional-hosting.com` und
`ns8.professional-hosting.com` **out-of-bailiwick** — sie liegen in
einem anderen Domain-Baum. Resolver folgen normaler Recursion ueber
die `.com`-Registry und finden die A/AAAA aus 3a.

**Glue-Records beim Customer-Domain-Registrar (`.de`, `.org`, ...)
sind nicht erforderlich** und werden dort meist gar nicht
unterstuetzt. Wer dort "Hostnames"/"Glue Records" sucht, sucht
vergeblich — fuer den Customer-Domain-NS-Switch ist nur die
NS-Delegation noetig (Schritt 7).

### 3c) Sonderfall: `professional-hosting.com` selbst auf ns8/ns1 hosten

NUR wenn die Provider-Domain `professional-hosting.com` direkt auf
dieser Infrastruktur (ns8 als Master, ns1 als Mirror) gehostet werden
soll, entsteht ein Henne-Ei-Problem: die A/AAAA fuer `ns1`/`ns8`
stehen in einer Zone, die ohne `ns1`/`ns8` gar nicht aufloesbar ist.

In dem Fall braucht es zusaetzlich **Glue-Records bei der
`.com`-Registry** (NICHT bei jeder Customer-Domain — nur einmalig fuer
die Provider-Domain):

| Hostname                       | IPv4            | IPv6                    |
|--------------------------------|-----------------|-------------------------|
| `ns1.professional-hosting.com` | `78.46.253.197` | `2a01:4f8:c014:8177::1` |
| `ns8.professional-hosting.com` | `178.63.43.203` | `2a01:4f8:120:538f::2`  |

UI-Pfad: bei Hetzner Robot `Domains -> professional-hosting.com ->
Nameservers -> Add own nameserver`; bei INWX/united-domains/Cloudflare
Registrar heisst die Funktion meist "Hostnames" oder "Child
Nameservers".

> **Architekturentscheidung:** Wenn `professional-hosting.com`
> stattdessen auf externem DNS bleibt, entfaellt dieser Schritt
> komplett — nur 3a ist relevant.

## Schritt 4 - Template auf bestehende Domains anwenden

Sobald Schritt 1 fertig ist, zeigt Plesk oben den gelben Banner:

> "DNS-Zonen-Template wurde geaendert. Alle neu erstellten Domains werden
> dieses Template fuer ihre DNS-Zonen anwenden. Klicken Sie auf hier, um
> die Aenderungen auf die bestehenden Domains anzuwenden."

**Auf den `hier`-Link klicken.** Ohne diesen Schritt behalten bereits
auf `ns8` existierende Zonen ihr altes SOA mit MNAME=`ns1` -> Secondary
lehnt NOTIFYs fuer sie ab (kein matching supermaster, weil supermasters
nach Schritt 5 auf MNAME=`ns8` eingestellt wird).

Selektive Alternative pro Domain:
`Domains -> <domain> -> DNS-Einstellungen -> Standard-Zonen-Records
wiederherstellen`.

Verifikation:

```bash
dig @178.63.43.203 aschenbrenner-bau.de SOA +short
# Erstes Token muss jetzt ns8.professional-hosting.com. sein (NICHT mehr ns1.)
```

## Schritt 5 - Secondary konfigurieren (`dns-admin primary add`)

Auf dem Secondary `ns1.professional-hosting.com` einloggen.

### 5a) MNAME-Check (Selbstkontrolle)

```bash
sudo dns-admin primary discover 178.63.43.203 aschenbrenner-bau.de
```

Erwartete Ausgabe NACH Schritt 4:

```text
============================================
  Discovery-Ergebnis
============================================
  Zone:    aschenbrenner-bau.de
  Primary: 178.63.43.203
  MNAME:   ns8.professional-hosting.com
  Status:  NICHT in supermasters - dieser MNAME wird beim NOTIFY abgewiesen!

  Eintrag hinzufuegen mit:
    sudo dns-admin primary add ns8.professional-hosting.com 178.63.43.203
```

Wenn der MNAME hier noch `ns1.professional-hosting.com` lautet ->
Schritt 1c oder Schritt 4 unvollstaendig. Erst dort fixen, dann hier
weiter.

### 5b) Primary `ns8` aufnehmen

```bash
sudo dns-admin primary add ns8.professional-hosting.com \
  178.63.43.203 2a01:4f8:120:538f::2
```

Das schreibt **zwei** Eintraege in `supermasters`:

| ip                     | nameserver                     |
|------------------------|--------------------------------|
| `178.63.43.203`        | `ns8.professional-hosting.com` |
| `2a01:4f8:120:538f::2` | `ns8.professional-hosting.com` |

PowerDNS wird automatisch neu geladen.

### 5c) Pruefen

```bash
sudo dns-admin primary list
```

Beide `ns8`-Zeilen muessen erscheinen — neben den bereits vorhandenen
`ns2`..`ns7`-Eintraegen, die durch dieses Setup nicht angefasst werden.

## Schritt 6 - Funktion verifizieren

### 6a) NOTIFY ausloesen

Auf `ns8` (Plesk) eine beliebige Zone minimal anfassen (z.B. TTL eines
Records auf 3601 setzen + Speichern) — oder warten, bis Plesk durch das
neu eingetragene Slave-Setup automatisch alle Zonen pusht.

### 6b) Live-Log auf dem Secondary (`ns1`)

```bash
sudo journalctl -u pdns -f
```

Erwartet:

```text
Received NOTIFY for aschenbrenner-bau.de from 178.63.43.203
AXFR done for aschenbrenner-bau.de (22 records)
```

### 6c) Zonen-Bestand pruefen

```bash
sudo dns-admin zone list | grep -i aschenbrenner
```

Die Zonen von `ns8` muessen jetzt zusaetzlich zu den bestehenden Zonen
(von `ns2`..`ns7`) im Inventory erscheinen.

### 6d) Autoritative Antwort vom Secondary

```bash
# Direkt am Secondary anfragen
dig @<ipv4-ns1> aschenbrenner-bau.de SOA
dig @<ipv6-ns1> aschenbrenner-bau.de SOA

# AA-Flag muss gesetzt sein
# MNAME im SOA muss ns8.professional-hosting.com. lauten (NICHT ns1.)
```

### 6e) Health-Check

```bash
sudo dns-admin health
```

Exit-Code 0 (`OK`) -> Aufbau produktiv.

## Schritt 7 - NS-Delegation am Registrar umstellen

Bis hierhin antwortet `ns1` autoritativ fuer alle gespiegelten Zonen,
aber **die Welt fragt ihn noch nicht**: die `.com`-Registry verweist
fuer `professional-hosting.com` noch auf die alten externen
Nameserver. Erst dieser Schritt aktiviert den neuen Aufbau extern.

> **Voraussetzung:** Schritt 6 muss erfolgreich durchgelaufen sein —
> autoritative Antworten von `ns1`, Health-Check `OK`. Wer hier
> umschaltet, bevor `ns1` antwortet, killt kurzzeitig den DNS-Dienst
> der Domain.

### 7a) Vor dem Umschalten: aktuelle Delegation pruefen

```bash
# Welche NS sind aktuell bei .com fuer professional-hosting.com hinterlegt?
dig +short NS professional-hosting.com @a.gtld-servers.net

# Erwartete Ausgabe vor dem Umschalten: externe NS
# (z.B. ns1.hetzner.de, ns2.hetzner.de o.ae.)
```

Das Ergebnis dieser Abfrage ist die **delegierte NS-Liste** auf
TLD-Ebene — sie wird gleich ersetzt.

### 7b) Im Registrar-Backend die NS-Records ersetzen

Die Domain `professional-hosting.com` ist aktuell auf externe
Nameserver delegiert. Diese externen NS muessen durch `ns1` + `ns8`
ersetzt werden. Wo das passiert, haengt vom Registrar ab:

| Registrar            | Typischer Pfad                                         |
|----------------------|--------------------------------------------------------|
| Hetzner Robot        | `Domains -> professional-hosting.com -> Nameservers`   |
| INWX                 | `Domain-Details -> Nameserver-Konfiguration`           |
| united-domains       | `Domain-Verwaltung -> Nameserver`                      |
| Cloudflare Registrar | `Domain -> DNS -> Nameservers`                         |
| OVH                  | `Domain -> DNS-Server`                                 |

Die aktuell hinterlegten externen Nameserver entfernen und durch
folgende ersetzen:

```text
Nameserver 1:  ns1.professional-hosting.com
Nameserver 2:  ns8.professional-hosting.com
```

> Mehr Slots im Registrar (typisch 4-8 erlaubt)? Leer lassen ODER mit
> spaeter folgenden Plesks (`ns9`, `ns10`, ...) auffuellen, sobald
> diese existieren. Zwei NS reichen technisch, mehr geben Resilienz.

### 7c) Propagation pruefen

Sofort nach dem Speichern sieht **nur** die `.com`-Registry die neue
Liste. Resolver, die noch eine alte NS-Antwort gecacht haben, kommen
erst nach Ablauf der parent TTL (`.com` typischerweise 86400s = 24h)
um.

```bash
# Bei .com-Registry direkt fragen (umgeht alle Caches):
dig +short NS professional-hosting.com @a.gtld-servers.net

# Erwartet: ns1.professional-hosting.com. + ns8.professional-hosting.com.
```

Externe Propagation tracken (24-48h Geduld):

```bash
# Mehrere offene Resolver befragen:
for r in 1.1.1.1 8.8.8.8 9.9.9.9; do
  echo "--- $r ---"
  dig @$r NS professional-hosting.com +short
done
```

Wenn alle drei die neue Liste zeigen, ist die Welt umgestellt.

### 7d) Kundendomains analog

Fuer jede auf `ns8` gehostete **Kundendomain** (`aschenbrenner-bau.de`,
weitere) muss separat — beim jeweiligen Registrar der Domain — die
NS-Delegation auf `ns1` + `ns8` umgestellt werden. Das ist nicht Teil
dieses ns8-Setups, sondern eine Pro-Kunde-Migration:

1. Beim Registrar der Kundendomain einloggen (kann ein anderer als der
   von `professional-hosting.com` sein).
2. Aktuelle NS-Liste durch `ns1.professional-hosting.com` +
   `ns8.professional-hosting.com` ersetzen.
3. 24-48h Propagation abwarten.

> **Pre-Migration-Check:** Vor dem Umstellen einer Kundendomain
> verifizieren, dass die Zone auf `ns8` (Plesk) und `ns1` (Secondary)
> identisch antwortet:
>
> ```bash
> dig @178.63.43.203 <kundendomain> SOA +short    # Plesk direkt
> dig @<ipv4-ns1>    <kundendomain> SOA +short    # Secondary
> # Beide muessen identische SOA-Serial liefern.
> ```

## Erwartete Fehlerbilder

### Plesk-SOA zeigt nach Schritt 1c immer noch `ns1.*` als MNAME

- _Ursache:_ Template-Aenderung nicht gespeichert oder Schritt 4 nicht ausgefuehrt.
- _Loesung:_ Schritt 1c erneut speichern -> Schritt 4 (`hier`-Link) klicken -> mit `dig @<plesk-ip> <zone> SOA` verifizieren.

### `Received NOTIFY ... no matching supermaster` im pdns-Journal

- _Ursache:_ MNAME im SOA passt nicht zum Nameserver-Wert im `supermasters`-Eintrag. Typisch nach Schritt 1c, wenn Schritt 4 vergessen wurde (Plesk schickt fuer alte Zonen noch SOA mit MNAME=`ns1`, supermasters erwartet `ns8`).
- _Loesung:_ `dns-admin primary discover 178.63.43.203 <zone>` -> MNAME ablesen -> Plesk-SOA-Template und supermasters synchron bringen.

### AXFR-Fehlschlag (`Transfer failed`), NOTIFY kam aber an

- _Ursache:_ `allow-transfer` auf Plesk fehlt fuer `ns1`.
- _Loesung:_ Schritt 2 wiederholen — `ns1` muss beidseitig (v4 + v6) als Slave eingetragen sein.

### `dig @<ns1-ip> ... SOA` ohne AA-Flag

- _Ursache:_ Anfrage geht nicht an den Secondary, sondern an einen Upstream-Resolver.
- _Loesung:_ `@<ns1-ip>` im dig-Aufruf explizit pruefen (kein Tippfehler bei der IP).

### Externe Resolver finden `ns8` nicht (`SERVFAIL`)

- _Ursache:_ Glue-Records bei `.com`-Registrar fuer `ns8` fehlen.
- _Loesung:_ Schritt 3a fuer `ns8` nachholen, 24-48h Propagation abwarten.

### Plesk speichert MNAME ohne trailing dot in der UI

- _Ursache:_ Plesk haengt intern den dot bei der Zonenerzeugung an.
- _Loesung:_ Ignorierbar — der dot ist in der Persistenz egal.

### Nur einige Zonen von `ns8` kommen durch

- _Ursache:_ Pro-Zone-Override in Plesk ueberschreibt das Template (alte `DNS Settings` der Zone).
- _Loesung:_ Bei den betroffenen Zonen: `DNS Settings -> Restore default zone records`.

## Spaeter: Weitere Primaries (`ns9`, `ns10`, ...) anbinden

Wenn der naechste Plesk-Server dazukommt, ist der Vorgang identisch —
nur die Werte tauschen:

1. Neuen Plesk installieren, FQDN `ns<N>.professional-hosting.com` setzen.
2. Auf dem neuen Plesk: Schritte 1-4
   - SOA-MNAME = `ns<N>.professional-hosting.com.` (eigener Hostname,
     nicht `ns1`)
   - NS-Records = `ns<N>` + `ns1`
   - Slave-DNS-Server = `ns1` (v4 + v6)
   - Glue im Registrar fuer `ns<N>` setzen, Template anwenden.
3. Auf `ns1` (Secondary):

   ```bash
   sudo dns-admin primary add ns<N>.professional-hosting.com \
     <ipv4-ns<N>> <ipv6-ns<N>>
   ```

4. Schritt 6 zur Verifikation.

**Geht ein Plesk-Primary spaeter offline** (Migration, Decom):

```bash
# Auf ns1:
sudo dns-admin primary remove ns<N>.professional-hosting.com
sudo dns-admin zone list                          # verwaiste Zonen identifizieren
sudo dns-admin zone delete <verwaiste-zone>       # ggf. einzeln aufraeumen
```

Der Secondary `ns1` selbst bleibt bei diesen Operationen unangetastet
— das ist der Sinn des `n:1`-Patterns.
